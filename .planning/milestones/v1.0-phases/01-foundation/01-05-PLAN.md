Create `src/lib/alerts.ts` with a `checkActionThresholds` function that:

1. Accepts a Supabase client (the service_role client from the cron) and a correlationId string.
2. Queries `job_logs` for action-type entries from the last 1 hour:
   ```sql
   SELECT status, COUNT(*)::int as count
   FROM job_logs
   WHERE job_type = 'action'
     AND finished_at > NOW() - INTERVAL '1 hour'
   GROUP BY status
   ```
   Use the Supabase client's `.rpc()` or raw query. Since Supabase JS doesn't support raw SQL easily, use `.from("job_logs").select("status").eq("job_type", "action").gte("finished_at", oneHourAgo)` and aggregate in JS.
3. Calculate:
   - `totalActions` = total count of action job_logs in the window
   - `successCount` = count where status = 'completed'
   - `timeoutCount` = count where status = 'timeout'
   - `successRate` = totalActions > 0 ? (successCount / totalActions) * 100 : 100
   - `timeoutRate` = totalActions > 0 ? (timeoutCount / totalActions) * 100 : 0
4. If totalActions < 5, skip alerting (not enough data for meaningful rate).
5. If successRate < 80, call `Sentry.captureMessage("OBSV-04: Action success rate below 80%", { level: "warning", fingerprint: ["obsv04-low-success-rate"], extra: { successRate, totalActions, successCount, timeoutCount, correlationId } })` and `logger.warn(...)`.
6. If timeoutRate > 5, call `Sentry.captureMessage("OBSV-04: Action timeout rate above 5%", { level: "warning", fingerprint: ["obsv04-high-timeout-rate"], extra: { timeoutRate, totalActions, timeoutCount, correlationId } })` and `logger.warn(...)`.
7. Return `{ totalActions, successRate, timeoutRate, alertsFired: string[] }` for the cron to include in its response.

Import Sentry from `@sentry/nextjs` and logger from `@/lib/logger`. The function must be async. Export it as a named export.

Also create `scripts/sentry-alert-rules.ts` — a standalone script (run with `npx tsx scripts/sentry-alert-rules.ts`) that uses the Sentry API to create two alert rules:

1. Read SENTRY_AUTH_TOKEN, SENTRY_ORG, SENTRY_PROJECT from env vars.
2. Use `fetch` to POST to `https://sentry.io/api/0/projects/{org}/{project}/rules/` with:
   - Rule 1: name "OBSV-04: Low Action Success Rate", conditions: event with message containing "obsv04-low-success-rate" seen more than 1 time in 10 minutes, action: send notification to all project members via email.
   - Rule 2: name "OBSV-04: High Timeout Rate", conditions: event with message containing "obsv04-high-timeout-rate" seen more than 1 time in 10 minutes, action: send notification to all project members via email.
3. Log success/failure for each rule creation.

The script uses Sentry's Issue Alert Rules API format:
```json
{
  "name": "OBSV-04: Low Action Success Rate",
  "conditions": [
    {"id": "sentry.rules.conditions.event_frequency.EventFrequencyCondition", "value": 1, "interval": "10m"}
  ],
  "filters": [
    {"id": "sentry.rules.filters.tagged_event.TaggedEventFilter", "key": "fingerprint", "match": "eq", "value": "obsv04-low-success-rate"}
  ],
  "actions": [
    {"id": "sentry.mail.actions.NotifyEmailAction", "targetType": "IssueOwners"}
  ],
  "actionMatch": "all",
  "filterMatch": "all",
  "frequency": 60
}
```
