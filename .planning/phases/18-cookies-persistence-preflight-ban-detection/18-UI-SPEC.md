# Phase 18 â€” UI Design Contract

> Visual and interaction contract for the Account-Degraded Banner, the Reconnect button, two new health-status badge variants, and three account-warning email variants. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Phase 18 ships almost no new UI. It extends three existing surfaces:

1. **Dashboard layout** gets a server-component banner above the main content when any account is degraded (`src/components/account-degraded-banner.tsx`).
2. **Account card** gets a "Reconnect" button when status is `needs_reconnect` or `captcha_required` (extends `src/features/accounts/components/account-card.tsx`).
3. **Health badge** adds two new state variants (`needs_reconnect`, `captcha_required`) following the existing inline-tint pattern (`src/features/accounts/components/health-badge.tsx`).
4. **Email template** in `send-account-warning.ts` branches three subject/body variants.

Memory rules honored: never expose "proxy" / "profile" / "fingerprint" terminology; no credit-burn math; sentence case throughout; native shadcn first.

## Copywriting Contract

All copy is sentence case (memory rule: `feedback_landing_copy_style`). No exclamation marks. No "the problem your product solves" framing.

### Banner

| Element | Copy |
|---------|------|
| Heading (1+ degraded) | "Some accounts need attention" |
| Heading (only 1 degraded) | "1 account needs attention" |
| Body lead-in | "These accounts are paused until you fix them:" |
| Per-row format | `{platform icon} u/{handle}` (Reddit) / `{platform icon} {name}` (LinkedIn) â€” followed by status badge + action button |
| Empty state | n/a â€” banner is conditionally rendered; if list is empty the component returns `null` |

### Per-row status copy (right of badge, before button)

| Status | Inline reason copy | Button label |
|--------|--------------------|--------------|
| `banned` | "Account suspended on the platform." | "View" (links to account detail) |
| `warning` | "Recent failures â€” slowing down." | "View" |
| `cooldown` | "Cooling down after a failure." | "View" |
| `needs_reconnect` | "Logged out â€” please sign back in." | "Reconnect" |
| `captcha_required` | "Captcha is blocking actions." | "Reconnect" |

### Account-card extension

| Element | Copy |
|---------|------|
| New button label | "Reconnect" |
| Button visibility | Only when `health_status IN ('needs_reconnect','captcha_required')` |
| Button action | Anchor with `target="_blank" rel="noopener noreferrer"` to the existing GoLogin Cloud Browser URL (same URL the connect flow uses) |
| Button icon | Phosphor `ArrowSquareOut` (16px) on the right |

### Empty / non-degraded state

The banner does not render when no accounts are degraded. The account card simply hides the Reconnect button. No "all good" affordance â€” absence is the success signal.

### Error state (banner)

If the underlying query in `(app)/layout.tsx` throws, the banner does not render (server component wraps the query in try/catch and returns `null`). Errors are reported to Sentry. No user-visible error UI in this phase â€” failures here would only mask other issues, never block actions.

### Destructive actions

None in this phase. The Reconnect flow is non-destructive (opens the cloud browser; user logs in or solves a captcha; next worker tick clears the status). No `<AlertDialog>` confirmations needed.

### Email copy (D-19, three reason branches)

The existing `send-account-warning.ts` helper extends to three subject/body variants. Plain text bodies, sentence case, no marketing pleasantries. Each ends with a single CTA link to the dashboard `/accounts` page.

| Reason | Subject | Body (plain text) |
|--------|---------|-------------------|
| `banned` | "Your {platform} account {@handle} was suspended" | "Heads up â€” {@handle} was suspended on {platform} and we've stopped sending actions through it.\n\nIf this is a mistake, you can appeal directly with {platform}. Otherwise, connect a different account to keep your campaigns running.\n\nView account â†’ {dashboardUrl}/accounts" |
| `needs_reconnect` | "Reconnect needed for {@handle}" | "{@handle} got logged out and we can't recover the session automatically. Open your dashboard and click Reconnect to sign back in â€” takes about a minute.\n\nReconnect â†’ {dashboardUrl}/accounts" |
| `captcha_required` | "Captcha is blocking {@handle} â€” quick fix" | "{platform} is showing a captcha for {@handle}, so we've paused its actions. Open your dashboard and click Reconnect to solve the captcha in the cloud browser.\n\nFix it â†’ {dashboardUrl}/accounts" |

Token rules:
- `{@handle}` renders as `u/handle` for Reddit, `name` for LinkedIn (no `@` prefix on LinkedIn).
- `{platform}` renders as "Reddit" or "LinkedIn".
- `{dashboardUrl}` is `process.env.NEXT_PUBLIC_SITE_URL` (or fallback `https://repco.ai`).
- Debounced 24h via the existing `job_logs kind='account_warning_email'` check â€” UI doesn't need to surface this; user just sees the email at most once per day per account.

## Accessibility

- Banner uses `<Alert>` which sets `role="alert"` (announces on render to assistive tech). Since the banner is server-rendered on every authenticated page load, the alert announces only on first render of an authenticated page where degradation is present â€” acceptable.
- Reconnect button is a real `<a>` (or `<Button asChild><a>...</a></Button>`) with `target="_blank" rel="noopener noreferrer"` and an `aria-label="Reconnect {@handle}"` so screen readers disambiguate when multiple Reconnect buttons exist in the banner.
- Status badge `aria-label` already follows the `Health status: {status}` pattern at `health-badge.tsx:52`. Extend identically for the two new states.
- Color is never the only signal â€” every tint is paired with a label and an icon (when used in the banner row).
