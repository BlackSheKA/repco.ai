# Phase 17.5 Context — Browser Profile Allocator (Browserbase)

## Why this phase exists

Phase 17 ("Residential Proxy + GoLogin Profile Allocator") was abandoned mid-execution after empirical UAT revealed three blocking issues:

1. **Cloud quota dead-end**: GoLogin Professional plan caps `maxParallelCloudLaunches` at 1. Counter got stuck after we deleted profiles mid-session, blocking all further cloud-browser launches without GoLogin support intervention.
2. **Per-slot pricing scales badly**: GoLogin charges per profile slot (10/100/300/1000) regardless of usage. 100-user MVP needs 100 slots = ~$300/mo Enterprise. SaaS-style provisioning isn't GoLogin's target.
3. **Geo coverage**: GoLogin's geolocation proxy supports only 5 regions (us/uk/de/ca/in) — half our `country-map.ts` 7-country whitelist (PL/FR/AU) silently fails with HTTP 400.

Spike on Browserbase ([research doc](../../research/browserbase-vs-gologin.md)) confirmed:
- 25 concurrent on $20/mo Developer plan (vs GoLogin 1 on $24-49/mo)
- Persistent contexts native — cookies/localStorage/IndexedDB auto-save across sessions
- Iframe-embeddable live view URL — login happens INSIDE our app
- 201 countries supported for residential proxy
- Reddit network-security check passes (T-Mobile residential IP, soft CAPTCHA instead of hard block)
- Official MCP server `@browserbasehq/mcp-server-browserbase`

## Inherited decisions (from Phase 17 — preserved)

These stay valid because they describe USER and BUSINESS rules, not vendor specifics:

- **D-01**: `country = "US"` hardcoded at every call site for now. Allocator accepts `country: SupportedCountry` arg; `connectAccount` always passes `"US"`. Future phase wires real country source.
- **D-02 (BPRX-06)**: Reuse rule = first match WHERE `user_id + country_code` AND not already used by an account on the requested platform. Order by `created_at ASC LIMIT 1`. Cross-platform same-user share a single `browser_profiles` row; same-platform second account creates new context.
- **D-09**: NO race lock. Button-disabled-on-submit is the only guard. Two concurrent allocations may create two contexts — accepted edge case (cheap on Browserbase, no slot quota).
- **D-10**: Best-effort rollback. If any step after `createContext` fails: delete the context. If social_account INSERT fails: delete browser_profile + Browserbase context. On reuse-path failure, the existing context survives (still in use by other accounts).
- **D-11**: User-facing copy is exactly `"Could not set up the account right now — please try again in a moment."` Never expose vendor names or HTTP errors.
- **D-12 (BPRX-05)**: 7 supported countries: US, GB, DE, PL, FR, CA, AU. `country-map.ts` keeps the canonical (timezone, locale, userAgent, language) tuples.
- **D-13**: No Postgres CHECK on country code. Type union + helper throw are the only enforcement.
- **D-14**: Allocator lives in `src/features/browser-profiles/lib/allocator.ts`, function name `allocateBrowserProfile`. Return shape evolves (see D17.5-04).
- **D-16**: Mocked unit tests on the allocator deferred. UAT covers correctness. Pure-function unit tests (`country-map.ts`) keep their existing test suite.

### D17.5-06: Stagehand adoption — NOW for LinkedIn executors

User decision: adopt Stagehand (`@browserbasehq/stagehand`) for all 6 LinkedIn executors in this phase. Replace selectors with `page.act("click connect button")` and structured extraction with `page.extract({ schema: zod })`.

**Rationale**: LinkedIn DOM is repco's #1 maintenance burden. Phase 13 executor selectors broke 2× since shipping. Stagehand absorbs that volatility.

**Cost model**: Each `page.act` and `page.extract` = 1 LLM call. Use Claude Haiku 4.5 (cheap, fast, sufficient for click/extract). Estimated cost at scale: 50 users × 8 DMs/day × 30 days × ~3 actions/DM × $0.001 = ~$36/mc. Headroom for growth, accepted.

**Reddit inbox CU stays raw Haiku CU** (no Stagehand) — already works, no DOM-fragility there.

**Stagehand model config**: `model: "claude-haiku-4-5-20251001"`, `apiKey: process.env.ANTHROPIC_API_KEY`. Leverages existing `ANTHROPIC_API_KEY` — no new keys needed.

### D17.5-07: Session TTL policy

Per-operation tuning for cost optimization:

| Operation | timeout | keepAlive | Rationale |
|---|---|---|---|
| Connect/login flow (user-facing) | 1800s (30 min) | false | Allow time for MFA/CAPTCHA solving |
| Per-action (DM, Connect, Follow, Like, Comment) | 300s (5 min default) | false | Fits within action budget |
| Reddit inbox CU | 180s (3 min) | false | Page screenshot + Haiku call, fast |
| LinkedIn prescreen | 300s | false | Profile load + classify |

`keepAlive: false` everywhere — sessions end after their work, cookies auto-save to persistent context. Cheaper than holding sessions open all day.

## Threat model deltas from Phase 17

- **T-17.5-01** (Information Disclosure, Browserbase API key): same as before for any vendor key. Server-only, never echoed to client. Mitigation: env var, not in client bundle.
- **T-17.5-02** (Stagehand prompt injection via LinkedIn DOM): malicious LinkedIn page text could attempt to manipulate `page.act` instructions. Mitigation: Stagehand actions are scoped to single-action verbs (`click`, `type`, `extract`), not free-form. LinkedIn page text crosses into LLM context but only as element selection input.
- **T-17.5-03** (Iframe live view exfiltration): debuggerFullscreenUrl is unauthenticated by default — anyone with the URL can view the session. Mitigation: short-lived (session timeout), URL only generated server-side and passed to authenticated client. Don't log it; don't render in error toasts.
- Phase 17 threats T-17-02-* (handle injection, reuse cross-user, UNIQUE race) all carry over identically — algorithm is preserved.

## Open risks

- **Reddit CAPTCHA on first login** (observed in spike): user manually solves once, cookies persist via context. Mitigation: document in onboarding copy.
- **LinkedIn checkpoint flow** (manual phone/email verification): same handling — user solves once during connect, cookies persist.
- **Browserbase regional outage**: us-west-2 only initially. Future: add eu-central-1 fallback when EU customer arrives.
- **Stagehand ↔ Haiku reliability**: not battle-tested in our codebase. Plan bakes-in a fallback to Sonnet 4.6 per-executor on >2 failure rate.
