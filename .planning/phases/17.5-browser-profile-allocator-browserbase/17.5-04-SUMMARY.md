## Production bugs surfaced + fixed in this plan

### Bug A — `@browserbasehq/sdk@2.10.0` `contexts.delete()` always 400
SDK injects `Content-Type: application/json` on a body-less DELETE. BB API responds `400 "Body cannot be empty"`. Allocator's `.catch(() => {})` swallowed it silently. Every D-10 rollback in production was leaking one BB context per failed connect. Same for any user-initiated stop/cleanup path through `deleteContext`.
- Fix: `src/lib/browserbase/client.ts:deleteContext` uses raw `fetch` DELETE with `X-BB-API-Key` header only (no Content-Type). Returns 204 cleanly. E2E verified: create → delete → GET probe → 404.
- Test: `deleteContext throws when BROWSERBASE_API_KEY missing` added.

### Bug B — Stagehand v3 model format silently broke every LinkedIn action
`@browserbasehq/stagehand@3.3.0` rejects bare model names with `UnsupportedModelError` at `init()`. Both `worker.ts:379` and `linkedin-prescreen/route.ts:212` passed `modelName: "claude-haiku-4-5-20251001"`. The outer `try/catch` at `worker.ts:386-391` reported only `"Browserbase connection failed"`, masking the real cause.
- Real impact: every LinkedIn DM/Connect/Follow/Like/Comment + the prescreen cron silently failed at the same line in the same way, with no actionable telemetry. The `linkedin-${userId.slice(0,8)}` placeholder handles in production were a downstream symptom — prescreen never ran.
- Fix: both call sites use `"anthropic/claude-haiku-4-5-20251001"`. Typecheck + 90/90 vitest tests green.
- Memory note: `feedback_stagehand_v3_model_format.md` (added).

## Findings carried forward as Phase 17.9

UAT also surfaced two coupled bugs that are NOT in 17.5 scope:

- `verifyAccountSession` is a sham — writes `session_verified_at = now()` and returns `verified: true` UNCONDITIONALLY without ever opening a BB session or checking `li_at`/`reddit_session`.
- No auto-degrade — `health_status` stays `warmup` even after the platform revokes the auth cookie; "Session active" green badge stays lit indefinitely while the BB context is auth-walled.

Both are scoped into **Phase 17.9: Real Session Verification + Auto-Degrade** (added to ROADMAP, 4-plan sketch). Not blocking 17.5 ship — production worker.ts now actually runs (Bug B fix) and will emit `session_expired` failure modes that 17.9 will translate into the right UX state.

A separate observation also noted: `browserSettings.verified: true` (BB Advanced Stealth) returns `403 "Advanced stealth mode is only available on the Enterprise plan"` on Developer ($20/mo). One-line flip available in `client.ts` if/when we move to Enterprise.
