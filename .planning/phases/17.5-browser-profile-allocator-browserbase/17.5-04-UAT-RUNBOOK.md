### ⚠ Production finding — Browserbase SDK `contexts.delete` is broken

While building the orphan-check probe, discovered that `@browserbasehq/sdk@2.10.0`'s `bb.contexts.delete(id)` always returns:
```
400 BadRequestError: Body cannot be empty when content-type is set to 'application/json'
```
The SDK injects `Content-Type: application/json` on a body-less DELETE; the BB API server rejects it. Verified with two independent runs (a fresh single-context create+delete probe, and the geolocation script's cleanup loop).

A raw `fetch("https://api.browserbase.com/v1/contexts/{id}", { method: "DELETE", headers: { "X-BB-API-Key": ... } })` returns 204 cleanly.

**Production impact**: `src/lib/browserbase/client.ts:56-63` (`deleteContext`) wraps the SDK call. Its callers in `src/features/browser-profiles/lib/allocator.ts:125,189` use `.catch(() => {})` — so the 400 is silently swallowed, the function returns "ok", but the BB context is **never actually deleted**. Every D-10 rollback in production leaks one Browserbase context. Same applies to any user-initiated delete/cleanup path that funnels through `deleteContext`.

The UAT here passes because the runner uses a raw-fetch `rawDeleteContext()` workaround, NOT the production code path. Surfaced as a follow-up — see `## Follow-ups` below.

### ⚠ Finding #2 — Production worker.ts breaks on every LinkedIn action (FIXED)

`@browserbasehq/stagehand@3.3.0` rejects bare model names with `UnsupportedModelError`; requires `provider/model` format. Production worker.ts:379 and `linkedin-prescreen/route.ts:212` both pass `modelName: "claude-haiku-4-5-20251001"`. The outer `try/catch` at `worker.ts:386-391` swallows the error and reports `"Browserbase connection failed"` — masking the real cause.

Impact: **every LinkedIn action attempt in production fails silently at Stagehand init**. Same for the `/api/cron/linkedin-prescreen` cron route. This explains why the outsi.com user's LinkedIn handle is still the placeholder `linkedin-a855c3ab` — prescreen never ran.

Fix in this plan: changed both call sites to `"anthropic/claude-haiku-4-5-20251001"`. Typecheck + 90 vitest tests green.
