URL is unauthenticated but session-scoped; expires when session ends. Threat T-17.5-03 mitigation (don't log, don't error-toast) is correct.

## Browserbase REST Integration

**Package:** `@browserbasehq/sdk` (official TypeScript SDK; preferred over raw REST for typing).

**Auth:** API key in `BROWSERBASE_API_KEY` env var. SDK reads automatically when constructed with `new Browserbase({ apiKey })`. Server-only.

**Project ID:** `BROWSERBASE_PROJECT_ID` env var (CONTEXT.md D17.5-01: `8220f64d-131b-47bd-8202-d0eb2fb45003`).

**Operations needed for `src/lib/browserbase/client.ts`:**

| Op | SDK call | REST equivalent | Returns |
|---|---|---|---|
| Create persistent context | `bb.contexts.create({ projectId })` | `POST /v1/contexts` | `{ id, projectId }` |
| Delete context | `bb.contexts.delete(contextId)` | `DELETE /v1/contexts/:id` | 204 |
| Create session | `bb.sessions.create({ projectId, browserSettings: { context: { id, persist: true } }, proxies, timeout, keepAlive })` | `POST /v1/sessions` | `{ id, connectUrl, status }` |
| Get session debug URL | `bb.sessions.debug(sessionId)` | `GET /v1/sessions/:id/debug` | `{ debuggerUrl, debuggerFullscreenUrl, wsUrl, pages: [...] }` |
| Release session | `bb.sessions.update(sessionId, { status: "REQUEST_RELEASE" })` | `POST /v1/sessions/:id` | session object |

**Proxies (BPRX-03):**
```ts
proxies: [{
  type: "browserbase",
  geolocation: { country: "US" },  // ISO-2 uppercase, 201 supported (BPRX-05)
}]
```

**Error codes (observed via Browserbase docs):**
- `400` invalid proxy country / malformed payload
- `401` bad API key
- `402` plan quota exceeded (e.g., concurrent session cap on Developer plan = 25)
- `404` context/session not found
- `429` rate limit (rare — generous limits)

**Recommended client.ts shape (mirrors gologin/client.ts style):**

```ts
// src/lib/browserbase/client.ts
import Browserbase from "@browserbasehq/sdk"

let _client: Browserbase | undefined
function client(): Browserbase {
  if (!_client) {
    const apiKey = process.env.BROWSERBASE_API_KEY
    if (!apiKey) throw new Error("BROWSERBASE_API_KEY not set")
    _client = new Browserbase({ apiKey })
  }
  return _client
}

function projectId(): string {
  const id = process.env.BROWSERBASE_PROJECT_ID
  if (!id) throw new Error("BROWSERBASE_PROJECT_ID not set")
  return id
}

export async function createContext(): Promise<{ id: string }> {
  const ctx = await client().contexts.create({ projectId: projectId() })
  return { id: ctx.id }
}

export async function deleteContext(contextId: string): Promise<void> {
  try {
    await client().contexts.delete(contextId)
  } catch (err: unknown) {
    // Browserbase returns 404 on already-deleted; treat as success
    if (isNotFound(err)) return
    throw err
  }
}

export interface CreateSessionArgs {
  contextId: string
  country: SupportedCountry  // ISO-2 from country-map.ts
  timeoutSeconds: number     // D17.5-07
  keepAlive?: boolean        // default false
}

export interface BrowserbaseSession {
  id: string
  connectUrl: string  // wss:// CDP endpoint
}

export async function createSession(args: CreateSessionArgs): Promise<BrowserbaseSession> {
  const s = await client().sessions.create({
    projectId: projectId(),
    browserSettings: { context: { id: args.contextId, persist: true } },
    proxies: [{ type: "browserbase", geolocation: { country: args.country } }],
    timeout: args.timeoutSeconds,
    keepAlive: args.keepAlive ?? false,
  })
  return { id: s.id, connectUrl: s.connectUrl }
}

export async function getSessionDebugUrl(sessionId: string): Promise<string> {
  const d = await client().sessions.debug(sessionId)
  return d.debuggerFullscreenUrl
}

export async function releaseSession(sessionId: string): Promise<void> {
  try {
    await client().sessions.update(sessionId, {
      projectId: projectId(),
      status: "REQUEST_RELEASE",
    })
  } catch (err) {
    if (isNotFound(err)) return  // session already ended
    throw err
  }
}
```

## Stagehand Integration (LinkedIn executors only — D17.5-06)

**Package:** `@browserbasehq/stagehand` (depends on `@browserbasehq/sdk`, `@anthropic-ai/sdk`, `playwright-core`).

**Instantiation pattern** (with existing Browserbase session — DOES NOT spawn its own browser):

```ts
import { Stagehand } from "@browserbasehq/stagehand"

const stagehand = new Stagehand({
  env: "BROWSERBASE",
  apiKey: process.env.BROWSERBASE_API_KEY!,
  projectId: process.env.BROWSERBASE_PROJECT_ID!,
  browserbaseSessionID: session.id,  // attach to existing session
  modelName: "claude-haiku-4-5-20251001",
  modelClientOptions: {
    apiKey: process.env.ANTHROPIC_API_KEY!,
  },
  verbose: 0,  // 0=silent, 1=warn, 2=info, 3=debug
})
await stagehand.init()
const page = stagehand.page  // Playwright Page with .act/.extract/.observe
```

**Usage primitives** (used in 5 LinkedIn executors):

```ts
// Click via natural language (replaces brittle selector chains)
await page.act("Click the Connect button on this profile header")

// Structured extraction (replaces innerText scraping)
const result = await page.extract({
  instruction: "Get the connection request status",
  schema: z.object({
    sentSuccess: z.boolean(),
    errorMessage: z.string().nullable(),
  }),
})

// Observation (low-cost preflight before .act)
const candidates = await page.observe("Connect or Follow buttons in the profile header")
```

**Cost model per CONTEXT.md D17.5-06:** Haiku 4.5 ≈ $0.001/call avg. ~3 actions/DM × 50 users × 8 DMs/day × 30 = ~$36/mo.

**Failure modes:**
- `page.act` returns `{ success: false, message }` when LLM cannot identify a target. Caller must check and map to executor `failureMode`.
- DOM mutations during action: Stagehand's auto-retry (configurable, default 3) handles most transients.
- LLM timeout (>30s default): rare on Haiku, but should be wrapped in `try/catch` with fallback to deterministic Playwright path for the executor's hot path.

**Cleanup:** `await stagehand.close()` calls `releaseSession` on Browserbase + closes Playwright. Use in `finally`.

**Threat T-17.5-02 (prompt injection via DOM)**: Stagehand only passes selected element text to the LLM as context, not full page HTML. Mitigation: keep instructions to single verbs ("click", "fill", "extract X"). Don't pass user-controlled prompts (free-form messages) into `page.act` arguments — only into deterministic `page.fill(selector, message)` after the dialog opens.

## CDP Connection (Reddit CU + worker default path)

For Reddit CU and any executor NOT adopting Stagehand:

```ts
import { chromium, type Browser, type Page } from "playwright-core"

const session = await createSession({ contextId, country: "US", timeoutSeconds: 180 })
const browser: Browser = await chromium.connectOverCDP(session.connectUrl)
const context = browser.contexts()[0] ?? await browser.newContext()
const page: Page = context.pages()[0] ?? await context.newPage()
// ... run CU work ...
await browser.close()  // closes Playwright; Browserbase session ends on releaseSession or timeout
await releaseSession(session.id)  // explicit release frees concurrent slot immediately
```

**Lifecycle invariant:** Always `releaseSession` in `finally` even when `browser.close()` succeeds. `browser.close()` alone does NOT free Browserbase concurrency slot — only `releaseSession` or session timeout does.

## Persistent Context Semantics

Confirmed from Browserbase docs:
- Context = Chrome user-data-dir snapshot stored on Browserbase. Persists cookies, localStorage, IndexedDB, sessionStorage.
- Created via `contexts.create`, ID stable forever until `contexts.delete`.
- TTL on context: **none** (vs sessions which have `timeout`). Contexts persist indefinitely.
- Session attaches via `browserSettings.context.id` + `persist: true`. On session end, all writes (cookies, etc.) flush back to context atomically.
- Two sessions on same context concurrently: **Browserbase serializes** — second session waits for first to release. D-09 (no race lock) still holds because allocator never starts a session.

**Implication for D17.5-09 (Phase 18):** BPRX-07 cookies_jar save/restore IS fully replaced by persistent contexts. Phase 18 can drop that requirement (document, don't implement).

## Iframe Live View (debuggerFullscreenUrl)

**Shape:** `https://www.browserbase.com/devtools-internal-compiled/index.html?wss=...&ws=connect.browserbase.com/debug/{sessionId}/devtools/page/...`

**Auth model:** URL contains a session-scoped WSS token. Anyone with the URL can view/interact during session lifetime. Mitigations (T-17.5-03):
- Server-only generation; pass to client via authenticated server action return value
- Never log to console / Sentry / Axiom
- Never render in user-visible error toasts
- Session timeout (1800s for connect flow per D17.5-07) bounds exposure window

**Iframe sandbox** (per UI-SPEC.md): `sandbox="allow-same-origin allow-scripts allow-forms allow-popups"`. Confirmed sufficient for clicks + keystrokes + form submission to reach inside the embedded session. `allow-popups` needed for Google OAuth fallback flow (LinkedIn / Reddit "Sign in with Google" opens new tab).

**`fullscreen` variant vs base `debuggerUrl`:** The fullscreen variant hides Chrome DevTools chrome (CONTEXT calls `debuggerFullscreenUrl` — that's the right one for a clean user-facing UI).

Per CONTEXT.md D-16, mocked unit tests on the allocator are deferred. UAT covers correctness. This research preserves that decision.

**Testable invariants per requirement:**

### BPRX-03: Residential proxy attached
- **Invariant**: Every Browserbase session created by the allocator pipeline has `proxies: [{ type: "browserbase", geolocation: { country } }]` in its config.
- **Validation**: UAT scenario 1 — create account, start session, inspect outbound IP via in-iframe visit to `https://api.ipify.org`. IP must geolocate to the requested country (use `https://ipinfo.io/{ip}/json`).
- **Code-level proof**: Grep `src/lib/browserbase/client.ts` for `proxies:` literal in `createSession` body.

### BPRX-05: 7-country whitelist
- **Invariant**: `country-map.ts` exposes exactly US, GB, DE, PL, FR, CA, AU. Any other country code throws.
- **Validation**: Existing unit tests on country-map.ts pass after refit. New: parameterized UAT scenario 3 — for each country code, create a session and verify proxy IP geolocates to expected country.
- **Code-level proof**: `mapForCountry("XX")` throws for any non-whitelisted code.

### BPRX-06: Profile reuse rule (D-02)
- **Invariant**: For same user_id + country, second account on different platform reuses existing browser_profile row. Second account on same platform creates a new browser_profile.
- **Validation**: UAT scenario 4 — create Reddit account, then LinkedIn account → 1 row in `browser_profiles`, 2 in `social_accounts`. Then second LinkedIn → 2 rows in `browser_profiles`.
- **Code-level proof**: Allocator's `existingProfile` query at allocator.ts:88-99 preserved (only column rename).

### D-10: Best-effort rollback
- **Invariant**: If `social_accounts` INSERT fails after Browserbase context create, the context is deleted (no orphans on Browserbase side). On reuse-path failure, the existing context survives.
- **Validation**: UAT scenario 5 — inject a synthetic INSERT failure (RLS denial via misconfigured client) and verify Browserbase contexts list shows no orphan via `bb.contexts.list`.

### D17.5-07: Session TTL policy
- **Invariant**: Connect flow uses 1800s timeout; per-action 300s; Reddit CU 180s.
- **Validation**: Code-level — grep callers for explicit `timeoutSeconds:` literal at each call site. Static check, no runtime test needed.

### Iframe interaction (UI-SPEC + D17.5-05)
- **Invariant**: User can complete LinkedIn login (email + password + checkpoint) inside the embedded iframe and the resulting cookies persist to the next session.
- **Validation**: UAT scenario 6 — start connect flow, log into LinkedIn inside iframe, click "Verify session", reload page, click "Open browser" again → user lands logged-in (cookies persisted via context).

### Stagehand action success rate
- **Invariant**: For each LinkedIn executor, `page.act("click X")` succeeds on a known-good profile ≥95% of the time on UAT smoke run.
- **Validation**: UAT scenario per executor — observe pass/fail across 5 prospects each. Below threshold → fall back to Sonnet 4.6 per CONTEXT D17.5-06 contingency.

## Open Risks & Ambiguities

1. **Stagehand SDK version pinning**: Stagehand has had breaking changes in 2025 (API stabilized but minor versions still moving). Pin `@browserbasehq/stagehand` to a specific minor version (`^1.x.x` not `latest`). Plan should commit `pnpm-lock.yaml`.
2. **Concurrent session limit**: Developer plan = 25 concurrent. With 50 active users running 8 DMs/day, peak parallelism could exceed 25. Mitigation: action-worker's existing serialization (worker processes one action at a time per account). Cross-user concurrency = number of active workers, currently bounded by Vercel cron concurrency. Monitor in production.
3. **Stagehand cost overrun on retries**: 3 retries × 3 LLM calls/action = 9 calls worst case. Set `maxRetries: 2` to bound. ~$0.009/action worst case.
4. **Session debug URL same-origin**: Browserbase serves on `browserbase.com` domain. Iframe in repco app is cross-origin. Cookies set inside the iframe go to browserbase.com (their auth) NOT to repco.ai. This is fine (intended); but if the iframe ever needs to read repco state (it doesn't), CORS would block.
5. **CONTEXT.md decomposition vs reality**: CONTEXT proposes 4 plans but executor refit + Stagehand adoption is meatier than the description suggests. **Recommend split: 4 plans is fine but plan 3 (executor refit) is the largest. Acceptable.**
