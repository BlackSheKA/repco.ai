<objective>
Stop-the-bleeding fix for the **G-01 / WR-01** GoLogin quota leak diagnosed in `15-UAT.md` and recorded in `15-REVIEW.md`. `connectAccount` currently calls `createProfile()` (consumes a GoLogin cloud profile slot) on every Add Account click and every Retry, but never writes the returned id anywhere — `social_accounts.browser_profile_id` is hard-coded to `null`. Each click orphans a profile.

Per `15-CONTEXT.md` Out-of-scope clause ("Allocator logic / GoLogin REST calls — Phase 17"), `connectAccount` should not be touching GoLogin at all in Phase 15. Phase 17's allocator will own all GoLogin REST calls and rewrite this action end-to-end. Until then, "Add account" should insert a placeholder `social_accounts` row with `browser_profile_id: null` and let the existing `startAccountBrowser` null-handling path surface the "allocator hasn't shipped yet" message to the user.

Purpose: Prevent further GoLogin quota waste on the dev account during the Phase 15 → Phase 17 transition window. This is intentionally minimal — Phase 17 will rewrite `connectAccount` entirely.

Output: 1 production source file gutted of GoLogin calls, 2 caller sites adjusted so the UX still flows.
</objective>
