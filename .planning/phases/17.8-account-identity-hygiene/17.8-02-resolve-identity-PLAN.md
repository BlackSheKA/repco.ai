Implement the server-side identity resolver `resolveAccountIdentity(accountId)` that
navigates the live Browserbase session to a profile-me URL, runs Stagehand
`extract()` against a server-defined Zod schema, server-side strict-regex re-validates,
and writes `reddit_username` or `linkedin_slug` + `identity_status='resolved'` to
`social_accounts`. Wire as fire-and-forget call from `verifyAccountSession`.

Purpose: Implements D-05, D-06, D-07, D-09. Identity columns are NEVER user-written.

Output: Connecting an account in dev → AccountCard transitions from `pending` to
`resolved` with the extracted username within ~10s.

Create `src/features/accounts/server/resolve-account-identity.ts`. Use service-role Supabase client (server-only file). Function signature:

```ts
export async function resolveAccountIdentity(
  accountId: string,
  opts?: { forceFreshSession?: boolean },
): Promise<{ success: true } | { error: "session_unavailable" | "extract_failed" | "validation_failed" | "duplicate" | "db_error" }>
```

Strict regexes (server-side, NON-NEGOTIABLE per RESEARCH §1):
```ts
const REDDIT_USERNAME_RE = /^[A-Za-z0-9_]{3,20}$/
const LINKEDIN_SLUG_RE = /^[a-z0-9-]{3,60}$/
```

Flow:
1. Load account: `select id, platform, browser_profile_id, browser_profile(current_session_id, browserbase_context_id)` from social_accounts.
2. Determine session: try `retrieveSessionStatus(current_session_id)`; if RUNNING and not `forceFreshSession`, reuse. Else allocate via `createSession({ contextId: browserbase_context_id })`. If both fail → return `{error:"session_unavailable"}` and UPDATE `identity_status='failed'`.
3. Instantiate Stagehand (pattern from linkedin-connect-executor.ts). Wrap whole block in try/catch.
4. For Reddit: `await page.goto("https://www.reddit.com/user/me", { waitUntil: "domcontentloaded", timeout: 20_000 })`, then `extract` with literal prompt `"Extract the Reddit username from the current profile URL (the segment after /user/). Strip 'u/' prefix if present."` and schema `z.object({ reddit_username: z.string() })` (loose; strict regex below).
5. For LinkedIn: navigate `https://www.linkedin.com/in/me`, extract with literal prompt `"Extract the LinkedIn slug from the current profile URL (the segment after /in/)."` and schema `z.object({ linkedin_slug: z.string() })`.
6. Server-side: run extracted value through the strict regex. On failure → `identity_status='failed'`, log warn with the bad value, return `{error:"validation_failed"}`.
7. UPDATE `social_accounts SET reddit_username=$1, identity_status='resolved'` (or `linkedin_slug`). On Postgres `23505`: DELETE the row, call `deleteContext(browserbase_context_id)` if present, return `{error:"duplicate"}`.
8. On success: return `{success:true}`. (Caller is fire-and-forget; AccountCard re-renders via existing realtime channel on `social_accounts` UPDATE.)
9. `await stagehand.close()` in `finally`. `await logger.flush()` if any logger is used.

CRITICAL trust boundary: prompt strings and Zod schemas are LITERALS in this file. Never interpolate `accountId` or any user-derived value into the prompt or schema. (T-17.5-02.)

Then create `resolve-account-identity.test.ts` covering every row of RESEARCH §5 + the trust-boundary assertion. Mock Stagehand class with `vi.mock("@browserbasehq/stagehand")`. Mock `@/lib/browserbase/client`. Mock the supabase service-role client.
