## Reddit u/ prefix handling — PREMISE CHALLENGED

| Option | Description | Selected |
|--------|-------------|----------|
| Hard reject with explanatory message | Strict | |
| Silently strip and accept | Forgiving copy-paste | |
| Strip + visible normalization toast | Educational | |

**User's choice:** None of the above. User pushed back on the entire premise: "Po co my tu tak komplikujemy ten handle input? Przecież to ma być zwykła nazwa konta roboczego dla użytkownika, żeby on wiedział, jakie konto tam loguje. Może nazwać sobie jakkolwiek. My tego nie używamy nigdzie w żadnym wywołaniu systemowym ani nic takiego."

**Claude's response:** Surfaced the actual call sites where `handle` IS used as a system input today (Reddit preflight `/user/{handle}/about.json`, deeplinks, executor navigation, BB session metadata). Asked user to choose between: keep validation, pivot to display-only label + separate identity columns, or hybrid.

| Option | Description | Selected |
|--------|-------------|----------|

**Notes:** This is the load-bearing decision. The rest of the discussion redesigned the phase around it.

## Schema shape

| Option | Description | Selected |
|--------|-------------|----------|
| Two separate columns (reddit_username + linkedin_slug) | Typesafe, indexable | ✓ |

**Notes:** Codified as D-02. CHECK constraint enforces exactly-one-non-null per platform.

## Extract failure handling

| Option | Description | Selected |
|--------|-------------|----------|
| identity_status='pending' → 'failed' + retry button | Explicit user action, gates worker | ✓ |
| Auto-retry 3× in background | No UI, hides failures | |
| Block verify until extract finishes | Conflicts with fire-and-forget choice | |

**Notes:** Codified as D-08, D-09. New `identity_status` ENUM. Worker queries everywhere need `.eq('identity_status', 'resolved')` — flagged as planner checklist.
