## 1. Stagehand `extract()` Pattern

**Canonical signature** (verified from `src/lib/action-worker/actions/linkedin-connect-executor.ts:187`):

```ts
const verdict = await stagehand.extract(
  "Detect whether a connection invitation was successfully sent for this LinkedIn profile",
  z.object({
    sent: z.boolean(),
    errorMessage: z.string().nullable(),
  }),
  { page },
)
```

**Identity extraction shape:**

```ts
// Reddit
await page.goto("https://www.reddit.com/user/me", { waitUntil: "domcontentloaded", timeout: 20_000 })
// /user/me 302-redirects to /user/{username} when authenticated
const result = await stagehand.extract(
  "Extract the Reddit username from the current profile URL (the segment after /user/). Strip 'u/' prefix if present.",
  z.object({ reddit_username: z.string().regex(/^[A-Za-z0-9_-]{3,20}$/) }),
  { page },
)

// LinkedIn
await page.goto("https://www.linkedin.com/in/me", { waitUntil: "domcontentloaded", timeout: 20_000 })
const result = await stagehand.extract(
  "Extract the LinkedIn slug from the current profile URL (the segment after /in/).",
  z.object({ linkedin_slug: z.string().regex(/^[a-z0-9-]{3,100}$/) }),
  { page },
)
```

**Important:** `extract()`'s Zod schema is enforced by Stagehand (parses the LLM output). Server MUST re-validate the result against the strict regex (`^[A-Za-z0-9_]{3,20}$` for Reddit, `^[a-z0-9-]{3,60}$` for LinkedIn) before writing to DB — Stagehand's regex is loose to tolerate extraction noise; the strict check at the boundary is non-negotiable.

**Trust boundary T-17.5-02 holds by construction:** schema and prompt are server-defined; no user-controllable text is ever passed to `extract()` or `act()`. Identity extraction is the safest possible Stagehand invocation.

**Stagehand instantiation:** No central wrapper exists — every executor instantiates inline. Pattern (from `linkedin-connect-executor.ts`):

```ts
const stagehand = new Stagehand({
  env: "BROWSERBASE",
  apiKey: process.env.BROWSERBASE_API_KEY!,
  projectId: process.env.BROWSERBASE_PROJECT_ID!,
  browserbaseSessionID: bbSessionId,
  modelName: "anthropic/claude-haiku-4-5-20251001",  // memory: must be `provider/model`
  modelClientOptions: { apiKey: process.env.ANTHROPIC_API_KEY! },
})
await stagehand.init()
const page = stagehand.page
// ... use page + stagehand.extract / stagehand.act
await stagehand.close()
```

For `resolveAccountIdentity` we should extract this into a small helper `src/features/accounts/server/resolve-account-identity.ts` rather than duplicating wrapping code yet again.

## 2. Migration Plan (00029)

**Filename:** `supabase/migrations/00029_identity_split.sql` (next sequential after `00028_reddit_stagehand_mechanism_costs.sql`).

**ENUM placement:** New ENUM `identity_status` lives in **the new migration**, NOT a retroactive edit to `00001_enums.sql`. Sequential migrations are the project convention (memory + CLAUDE.md "sequential migrations `00001_`, `00002_`…"). Migrations are append-only.

**DDL skeleton:**

```sql
-- 00029_identity_split.sql
-- Phase 17.8: split social_accounts.handle into display_name +
-- platform-specific identity columns. All current users are test data
-- (memory rule); TRUNCATE is safe and clean.

BEGIN;

-- 1. ENUM (new migration, append-only convention)
CREATE TYPE identity_status AS ENUM ('pending', 'resolved', 'failed');

-- 2. Wipe existing rows. Phase 17.5-01 already TRUNCATE'd this table on
-- migration; existing dev/prod rows already need reconnection. CASCADE
-- removes downstream references (actions, prospects bound to accounts).
TRUNCATE TABLE social_accounts CASCADE;

-- 3. Drop the conflated column.
ALTER TABLE social_accounts DROP COLUMN handle;

-- 4. Add the four new columns.
ALTER TABLE social_accounts
  ADD COLUMN display_name TEXT NOT NULL,
  ADD COLUMN reddit_username TEXT,
  ADD COLUMN linkedin_slug TEXT,
  ADD COLUMN identity_status identity_status NOT NULL DEFAULT 'pending';

-- 5. CHECK: exactly one platform identity column populated, matching the
-- platform value. NULL identity is allowed only while identity_status != 'resolved'.
ALTER TABLE social_accounts ADD CONSTRAINT social_accounts_identity_matches_platform
  CHECK (
    (platform = 'reddit'   AND linkedin_slug   IS NULL)
    AND (platform = 'linkedin' AND reddit_username IS NULL)
  );
-- NOTE: planner: refine this CHECK — current draft has a logic bug
-- (the AND between two platform branches is wrong). Correct shape:
--   CHECK (
--     (platform = 'reddit'   AND linkedin_slug   IS NULL) OR
--     (platform = 'linkedin' AND reddit_username IS NULL)
--   )
-- Use the CORRECT form in the actual migration. Tested form is documented
-- in §Validation Architecture.

-- 6. Partial UNIQUE indexes (one identity per user per platform; NULLs allowed).
CREATE UNIQUE INDEX social_accounts_user_reddit_uidx
  ON social_accounts (user_id, reddit_username)
  WHERE reddit_username IS NOT NULL;

CREATE UNIQUE INDEX social_accounts_user_linkedin_uidx
  ON social_accounts (user_id, linkedin_slug)
  WHERE linkedin_slug IS NOT NULL;

-- 7. Index for worker gating queries (`identity_status = 'resolved'` is on
-- the hot path for warmup / DM dispatcher / prescreen / check-replies).
CREATE INDEX social_accounts_user_resolved_idx
  ON social_accounts (user_id, platform)
  WHERE identity_status = 'resolved';

COMMIT;
```

**Migration ordering rationale:** TRUNCATE FIRST (no row-level constraints to satisfy after), then DROP, then ADD with NOT NULL defaults safe because table is empty. CHECK + UNIQUE indexes added last so they don't fight INSERTs during ADD COLUMN.

**RLS:** No RLS changes needed — `social_accounts` already has RLS policies on `(auth.uid() = user_id)` from `00001_enums.sql` / earlier migrations. New columns inherit.

**TypeScript types regen:** Run `pnpm supabase gen types` (or whatever the project script is — verify in `package.json`) so `social_accounts.handle` reads stop type-checking. This forces the planner to find every call site at compile time, which is the cheapest possible way to enforce the refit.

## 4. Worker Gating Snippet

Add to every worker query that picks an account to do non-display work:

```ts
const { data: accounts } = await supabase
  .from("social_accounts")
  .select("id, platform, reddit_username, linkedin_slug, browser_profile_id, ...")
  .eq("user_id", userId)
  .eq("identity_status", "resolved")  // ← NEW: Phase 17.8 gating
  .eq("health_status", "healthy")
```

**Rationale:** an account with `identity_status = 'pending'` or `'failed'` has no trustworthy username/slug → any deeplink, preflight, or DM would point at the wrong account (or 404). Silently skipping is correct — AccountCard surfaces the failed-state banner so the user retries explicitly (D-09).

**Exception:** display-only queries (`(app)/page.tsx`, `(app)/accounts/page.tsx`, `account-card.tsx`, `agent-card.tsx`) MUST NOT add this filter — pending/failed accounts still appear in the UI so the user sees them and can retry.

## 5. `resolveAccountIdentity` — Failure Handling Catalog

The new server action lives at `src/features/accounts/server/resolve-account-identity.ts`. Failure modes and transitions:

| Failure mode | Detection | Action | Status transition |
|---|---|---|---|
| BB session gone (user closed live view) | `retrieveSessionStatus` returns non-RUNNING | Allocate fresh session bound to `browser_profile.browserbase_context_id` (D-07 fallback) | continue with fresh session |
| Stagehand `init()` throws | try/catch around init | Log warn, transition to `failed` | `pending → failed` |
| Page nav fails (timeout / 4xx redirect) | try/catch around `page.goto` | Log warn, transition to `failed` | `pending → failed` |
| `extract()` throws | try/catch | Log warn, transition to `failed` | `pending → failed` |
| `extract()` returns malformed/empty value | result is null/undefined OR Zod schema fails | Log warn, transition to `failed` | `pending → failed` |
| Extracted value fails strict regex | Server-side `safeParse` of strict regex | Log warn with the bad value, transition to `failed` | `pending → failed` |
| UNIQUE constraint hit (account already exists for this user) | DB error code `23505` on UPDATE | Surface "to konto jest już połączone", DELETE the new `social_accounts` row, `deleteContext(browserbase_context_id)` | row deleted; user toast |
| Successful extract + write | UPDATE succeeds | `revalidatePath("/accounts")` | `pending → resolved` |

**No retry loop** (D-09). User retries explicitly via the "Spróbuj jeszcze raz" button on the failed banner. The retry button calls `resolveAccountIdentity(accountId)` with `forceFreshSession: true`.

**Trust boundary:** every `extract()` call uses the server-defined prompt + schema. No user input is ever inlined.

## 9. Validation Architecture

> Required for Nyquist VALIDATION.md generation (Step 5.5 of plan-phase workflow).

This phase has **5 validation dimensions** that must be exercised before the phase passes:

### Dimension 1 — Schema regex correctness (server-side)

After Stagehand `extract()`, the server re-validates the result against a strict regex. Tests must cover:
- Valid Reddit username `kamil_wandtke` → passes
- Reddit username with whitespace `"main repco acc"` → fails
- Reddit username starting with `u/` → fails (server strips prefix? or rejects? — planner picks; CONTEXT.md D-05 says strict regex is `^[A-Za-z0-9_]{3,20}$` so `u/` fails by length+slash)
- Reddit username 21+ chars → fails
- LinkedIn slug `john-doe-123` → passes
- LinkedIn slug `JohnDoe` → fails (uppercase)
- LinkedIn slug ending with `-` → fails
- Empty string → fails

### Dimension 2 — CHECK constraint behavior

Migration test cases (use Supabase Management API to dry-run):
- INSERT with `platform=reddit, reddit_username='x', linkedin_slug=NULL` → succeeds
- INSERT with `platform=reddit, reddit_username='x', linkedin_slug='y'` → fails (CHECK)
- INSERT with `platform=linkedin, reddit_username=NULL, linkedin_slug='x'` → succeeds
- INSERT with `platform=linkedin, reddit_username='x', linkedin_slug=NULL` → fails (CHECK)
- INSERT with `platform=reddit, reddit_username=NULL, linkedin_slug=NULL` → succeeds (pending state allowed)

### Dimension 3 — Worker gating SQL

For each worker (warmup, prescreen, check-replies, queue dispatch):
- Query returns row when `identity_status = 'resolved'`
- Query SKIPS row when `identity_status = 'pending'`
- Query SKIPS row when `identity_status = 'failed'`

### Dimension 4 — `resolveAccountIdentity` failure paths

For each row in §5 failure catalog: trigger the failure mode, assert the resulting `identity_status` value + side effects (row deletion, deleteContext call, toast).

### Dimension 5 — UNIQUE violation handling

Test the "to konto jest już połączone" path: connect an account, run extract, then connect a second account with the same identity → assert that the second row gets DELETE'd, BB context gets `deleteContext`'d, and the user sees the right toast.

### UAT (manual)

Two scenarios from CONTEXT.md SC #5/#6:
1. Connect Reddit → no input field → Live view → log in as `kamil_wandtke` → AccountCard shows `display_name` (auto-generated) primary, `u/kamil_wandtke` (real) secondary.
2. Connect LinkedIn → log in → AccountCard shows real `linkedin.com/in/{slug}`, NOT `linkedin-${id_prefix}`.
