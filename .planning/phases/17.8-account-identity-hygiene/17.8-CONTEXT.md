# Phase 17.8: Account Identity Hygiene - Context

**Gathered:** 2026-04-28
**Status:** Ready for planning — **SCOPE PIVOTED from ROADMAP entry**

<domain>
## Phase Boundary

**Pivot from the original ROADMAP entry.** The original framing — "validate user-supplied handle at connect, allow editing, auto-resolve LinkedIn handle" — treats `social_accounts.handle` as both a user-facing label AND a system input (used by Reddit preflight, deeplinks, executors). User pushback: "po co my komplikujemy ten handle input — to ma być zwykła nazwa konta roboczego dla użytkownika".

**New framing:** decouple display label from system identity entirely.

- **Display label** (`display_name`) — free-form text the user sees on AccountCard. No validation beyond non-empty + length cap. Editable post-connect with no system side effects.
- **Platform identity** (`reddit_username`, `linkedin_slug`) — **only ever written by the server, never by user input.** Auto-extracted via Stagehand `extract()` after successful login. Used by Reddit preflight, deeplinks, executor navigation, BB session metadata, and DM/approval-queue rows.

This phase delivers the schema migration, extraction server actions, connect-flow simplification (no handle input), and call-site refit so that platform identity is always trustworthy.

**Out of scope:**
- Handle uniqueness across users (only enforced per-user-per-platform)
- Reddit `u/` prefix UX (moot — no Reddit input field exists anymore)
- Edit modal with Zod validation against platform schemas (the original SC #3 — replaced by simple `updateDisplayName` since display_name has no system constraints)

</domain>

<decisions>
## Implementation Decisions

### Architecture pivot
- **D-01: Handle is split into two concepts.** Drop `social_accounts.handle` (the conflated column) entirely. Add `display_name` (user-facing label) and platform-specific identity columns (`reddit_username`, `linkedin_slug`). Identity columns are server-only writes.
- **D-02: Platform identity uses two typed columns, not one polymorphic.** `social_accounts.reddit_username TEXT` and `social_accounts.linkedin_slug TEXT`. Exactly one is non-NULL per row, enforced by a `CHECK (CASE platform WHEN 'reddit' THEN reddit_username IS NOT NULL AND linkedin_slug IS NULL ...)` constraint. Plus typesafe and indexable; a future third platform pays an `ALTER TABLE`, which is acceptable.
- **D-03: Uniqueness enforced at DB level.** `UNIQUE (user_id, platform, reddit_username) WHERE reddit_username IS NOT NULL` and analogous for `linkedin_slug`. On extract, if the constraint is hit, surface "to konto jest już połączone", delete the new `social_accounts` row, and `deleteContext` its BB context.

### Connect flow
- **D-04: No handle input on the connect screen.** ConnectionFlow drops the input entirely. User clicks `Connect Reddit` or `Connect LinkedIn` and is taken straight into BB live view. `connectAccount(platform)` no longer takes a `handle` arg. `display_name` is auto-populated from the extracted username after verify (or from a fallback like `Reddit konto 1` if extract is still pending — see D-08 for state handling).

### Identity extraction
- **D-05: Stagehand `extract()` for both platforms.** LinkedIn already in scope (SC #4). Reddit gets the same mechanism for consistency (rejected: `/api/v1/me.json` despite being cheaper/deterministic). Schema: `z.object({ reddit_username: z.string() })` / `z.object({ linkedin_slug: z.string() })`. Server validates extracted value against the strict regex (`^[A-Za-z0-9_]{3,20}$` for Reddit, `^[a-z0-9-]{3,60}$` for LinkedIn) before writing — drift in the LLM output never reaches the DB.
- **D-06: Extraction is fire-and-forget after `verifyAccountSession`.** `verifyAccountSession` returns immediately as it does today; `resolveAccountIdentity(accountId)` is invoked unawaited in the same request. AccountCard re-renders via the existing realtime channel when the column changes. T-17.5-02 trust boundary preserved (`extract()` schema is server-defined; no user text is concatenated into `act()` args).
- **D-07: Reuse the live-view BB session.** `resolveAccountIdentity` reads `social_accounts.browser_profile.current_session_id`, calls `retrieveSessionStatus`, and reuses if RUNNING. If the session is gone (user closed live view before clicking verify), fall back to allocating a fresh session bound to `browserbase_context_id` (Phase 17.5 cookie persistence makes this safe).

### State machine + failure handling
- **D-08: New column `social_accounts.identity_status` ENUM.** Values: `pending` (default after connect), `resolved` (extract wrote a valid value), `failed` (extract gave up). Workers (warmup, DM, intent feed) MUST gate on `identity_status = 'resolved'` — accounts in `pending`/`failed` are skipped silently (deeplinks would point at the wrong account otherwise). AccountCard shows a degraded banner + explicit `Spróbuj jeszcze raz` button when status is `failed`.
- **D-09: No automatic retry loop.** First extract attempt = inside the post-verify request. Failure transitions to `failed`. User retries explicitly via the button (which spawns a fresh BB session bound to `browserbase_context_id` and re-runs `resolveAccountIdentity`). No exponential-backoff worker job — keeps the failure surface visible.

### Edit affordance
- **D-10: Edit edits `display_name` only.** Pencil icon on AccountCard opens a small inline editor or modal (planner choice — both are fine since validation is trivial). New server action `updateDisplayName(accountId, displayName)`: trims, length-caps at 64 chars, rejects empty after trim, calls `revalidatePath("/accounts")`. No Zod schema needed beyond that. Identity columns are read-only from the UI.

### Backfill + migration safety
- **D-11: TRUNCATE existing `social_accounts` in the migration.** All current users are test data (memory rule). Existing rows already need reconnection after the Phase 17.5-01 Browserbase context migration (which already TRUNCATE'd CASCADE). One more TRUNCATE in this phase is a no-op for prod and clean for dev. Migration order: `TRUNCATE social_accounts CASCADE` → `ALTER TABLE` (drop `handle`, add `display_name`, `reddit_username`, `linkedin_slug`, `identity_status`) → add unique partial indexes + CHECK constraint.

### Claude's Discretion
- Inline editor vs modal for `updateDisplayName` — planner picks. Both are simple.
- Exact CHECK constraint syntax (single CHECK with CASE vs platform-specific partial CHECKs) — planner picks; semantics are clear.
- `display_name` default while `identity_status = 'pending'`: either auto-generated (`Reddit konto N`) or empty until extract resolves. Planner picks based on AccountCard rendering needs.

</decisions>

## Existing Code Insights

### Reusable Assets
- `verifyAccountSession` (account-actions.ts:346-372) — already returns immediately and revalidatePath's. Drop-in spot to fire `resolveAccountIdentity(accountId)` unawaited before returning.
- LinkedIn executors' Stagehand instantiation pattern — `resolveAccountIdentity` mirrors it (BB session + Stagehand wrap + `extract()`).
- Existing realtime channel on `social_accounts` (used by AccountCard for live status updates) — picks up `identity_status` and `reddit_username`/`linkedin_slug` column changes for free.
- Phase 17.5 `browser_profiles.browserbase_context_id` — fallback session allocation in D-07 binds to this.

### Established Patterns
- Server actions return `{ success, error }` shape; AccountCard uses `sonner` toasts.
- Migrations are sequential (`00025_…`, next is `00026_…`); RLS on every new table; `TIMESTAMPTZ DEFAULT now()`.
- ENUMs live in `00001_enums.sql` per CLAUDE.md convention (12 ENUMs there today; `identity_status` becomes the 13th).
- Trust boundary T-17.5-02: user-controllable text never inlined into `act()` args. Identity extraction is safe by construction (schema is server-defined; user has no control over `extract()` prompt or schema).

### Integration Points
- **Worker gating:** every place that picks an account to run a job (warmup scheduler, DM dispatcher, intent feed, prescreen cron) needs `.eq('identity_status', 'resolved')` added to its query. Easy to miss — flag for planner as a checklist.
- **Migration ordering:** TRUNCATE before ALTER TABLE; otherwise the `handle NOT NULL` constraint blocks the ALTER if any rows exist.
- **Phase 17.7 (in flight) is in scope-coordination range.** Reddit executor PLAN files read `social_accounts.handle` today. Either 17.7 ships first and 17.8 refits it, or 17.8 ships first and 17.7 reads `reddit_username` from day one. Planner: pick the order explicitly.

<specifics>
## Specific Ideas

- User explicitly framed display_name as "nazwa robocza dla użytkownika, żeby on wiedział, jakie konto tam loguje" — so the AccountCard should treat display_name as the primary visual identifier (large/bold) and the platform identity as secondary metadata (small/muted, with a link icon).
- "User klika 'Dodaj konto' i widzi ekran logowania" (memory: `feedback_no_proxy_ux_complexity.md`) — the connect flow simplification in D-04 aligns with this. Zero friction to start a connect.

</specifics>

<deferred>
## Deferred Ideas

- **`u/` prefix and `linkedin.com/in/` URL paste handling.** Originally a gray area — moot under the pivot since there's no user input field. If we ever add a power-user "I already know my username, skip extraction" path, this comes back.
- **Multiple-platform polymorphic identity (e.g., adding Twitter/X).** Two-column schema (D-02) pays an ALTER per new platform. Acceptable for now; revisit if/when a third platform is roadmapped.
- **Auto-retry loop with backoff for failed extractions.** Rejected (D-09) in favor of explicit user-triggered retry. If post-launch data shows users not noticing the `failed` banner, add a worker job then.
- **Cross-user uniqueness (one Reddit account = one user, globally).** Out of scope — uniqueness is per-user-per-platform only. If billing fraud or account-sharing becomes a problem, revisit.
- **Display name uniqueness validation.** Not enforced — two `display_name = "Praca"` rows for the same user are allowed (different platforms or just preference). No need to add complexity.

</deferred>
